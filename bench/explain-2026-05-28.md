# EXPLAIN findings — hot retrieval path (#44)

Date: 2026-05-28. Local Supabase Postgres 15 (`:54322`), migrated through 0004,
22,568 rows. Goal: prove + secure sub-100ms p95 on the hot path
(`api/internal/retrieve/postgres/repo.go`). Conclusion up front:

- **Every real (non-empty) hot query uses the GIN/GIST/btree indexes. No
  sequential scan on `businesses` on any production path.**
- **No index gap → no new migration added.** Adding a speculative index would
  be unjustified.
- Prepared statements: pgx default `QueryExecModeCacheStatement` (prepared on
  first use, cached per connection). 1s `statement_timeout` set at pool init
  plus a 1s `context.WithTimeout` in the adapter. Both correct, no change.
- Measured **p50=7ms, p95=25ms** (`go run ./cmd/bench-runner -generate 300
  -seed 42`, n=726, local DB).

## How measured

`EXPLAIN (ANALYZE, BUFFERS)` via
`docker exec supabase_db_lemon-search psql -U postgres -c "…"`. Wall-clock
latency taken separately with `\timing` (no ANALYZE instrumentation), since
ANALYZE inflates timing on the per-row `lemon_open_status` lateral (it is
called once per candidate row).

## 1. search_candidates — keyword query, empty overlay

`select * from search_candidates('coffee', 25.76, -80.19, now(), 150, null,
'{}','{}','{}','{}', false)`

```
Bitmap Heap Scan on businesses b  (rows=2521)
  BitmapOr
    Bitmap Index Scan on idx_biz_search_vec   Index Cond: (search_vector @@ '''coffe'''::tsquery)   rows=2521
    Bitmap Index Scan on idx_biz_name_trgm    Index Cond: (name % 'coffee'::text)                    rows=223
    Bitmap Index Scan on idx_biz_name_trgm    Index Cond: (name ~~* 'coffee%'::text)                 rows=189
  Function Scan on lemon_open_status os  (loops=2521)
```

- Indexes used: `idx_biz_search_vec` (GIN tsvector) + `idx_biz_name_trgm` (GIN
  trgm, twice: `%` similarity and `ilike` prefix), combined via `BitmapOr`.
- **No seq scan.** Bitmap heap scan finishes in ~4–6ms.
- Wall-clock (no ANALYZE overhead): ~51ms steady-state. "coffee" is the
  heaviest-recall term (2521 candidates); cost is dominated by the per-row
  `lemon_open_status` lateral + distance projection over the recall set, not by
  index access. Still well under the 100ms budget. Lighter terms checked:
  `tac` 67 rows, `x` 585, `sushi` 580, `barber` 838 — all index-driven, all
  faster.

## 2. search_candidates — overlay-narrowed (category + universal tag)

`select * from search_candidates('coffee', 25.76, -80.19, now(), 150,
'Food & Drinks', '{}', '{date-night}','{}','{}', false)`

```
Bitmap Heap Scan on businesses b  (rows=207)
  BitmapAnd
    Bitmap Index Scan on idx_biz_uni_tags   Index Cond: (universal_tags && '{date-night}')   rows=1958
    Bitmap Index Scan on idx_biz_category   Index Cond: (category = 'Food & Drinks')           rows=7710
    BitmapOr (idx_biz_search_vec + idx_biz_name_trgm ×2, as above)
  Function Scan on lemon_open_status os  (loops=207)
Execution Time: 13.7 ms (with ANALYZE)
```

- Overlay predicates are indexed and used: `idx_biz_uni_tags` (GIN array),
  `idx_biz_category` (btree), AND-combined with the text/trigram `BitmapOr`.
- **No seq scan.** Narrowing cuts the candidate set to 207, so the lateral runs
  207× instead of 2521× → faster than the unnarrowed case (~14ms even with
  ANALYZE overhead). Specific-tag / subcategory / price filters map to
  `idx_biz_spec_tags` / btree the same way.

## 3. ExactName

SQL in `repo.go` (`exactNameSQL`): trgm pre-filter `name % $1` then
`lemon_name_match($1, name) >= 0.8`, ordered, `limit 1`. Tested with a real
unique name "Shepherd Artisan Coffee":

```
Bitmap Heap Scan on businesses b  (rows=1)
  Bitmap Index Scan on idx_biz_name_trgm   Index Cond: (name % 'Shepherd Artisan Coffee')   rows=136
  Function Scan on lemon_open_status os  (loops=1)
Execution Time: 5.4 ms
```

- Index used: `idx_biz_name_trgm` (GIN trgm). The `%` pre-filter narrows to ~136
  rows; `lemon_name_match` then runs only on those, and the `lemon_open_status`
  lateral executes once (for the surviving row). For a non-matching query
  ("Starbucks") the lateral is `never executed`.
- **No seq scan.** ~5ms.

## Note: the empty-query seq scan is unreachable in production

`search_candidates('', …)` *does* seq-scan businesses (the
`p.q_clean is null OR …` recall clause matches all 22,568 rows — there is no
predicate to index, by design). This is **not** a production path: the API
short-circuits empty/whitespace-only queries before retrieval —
`internal/api/search.go:85` returns an empty `200 OK` when `q == ""` (after
`strings.TrimSpace`), so `repo.Search`/`ExactName` are never called with an
empty query. The bench-runner likewise only issues non-empty queries. No index
fixes this case and none is warranted.

## Prepared statements + statement timeout

- **Prepared statements** — `cmd/api/main.go` builds the pool with
  `pgxpool.ParseConfig` and does **not** override the query exec mode, so pgx
  uses its default `QueryExecModeCacheStatement`: each parameterized query
  (`searchSQL`, `exactNameSQL`) is prepared on first execution and the prepared
  statement is cached per connection and reused. All queries are parameterized
  (`$1…$11`); no string-concatenated SQL. Status: correct, no change.
- **Statement timeout** — `cmd/api/main.go:104` sets
  `cfg.ConnConfig.RuntimeParams["statement_timeout"] = "1000"` (1s) at pool
  init. The adapter additionally binds a 1s `context.WithTimeout(ctx, queryTO)`
  in both `Search` and `ExactName` (`repo.go:19`, `:105`, `:147`) so it stays
  correct regardless of pool wiring. Status: correct, no change.

## Index added?

**No.** All hot predicates are already covered (GIN on `search_vector`,
`name gin_trgm_ops`, `universal_tags`, `specific_tags`; btree on `category`,
`archetype`; GIST on `loc`) and the planner uses them. The GIST `idx_biz_loc`
intentionally does not appear in the recall plan: distance is a
projection/ordering term, not a filter (ADR-0003: recall must not hard-filter on
distance). No real gap exists, so no `0005` migration was created — consistent
with the "no speculative indexes" guidance.
